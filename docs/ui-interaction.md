# 0xRIP UI 交互规范

以下是 **0xRIP** 的完整交互系统规范，将"数字死亡"的体验渗透到每一个像素和每一次点击中。

---

## 一、核心交互隐喻

```
用户身份：探墓人（Tomb Visitor）
核心动作：探访（Visit）| 悼念（Mourn）| 招魂（Summon）— 与逝者重逢，一起讨论你们曾共同经历的事
```

---

## 二、导航系统：终端式指令

| 指令 | 动作 | 视觉反馈 |
|:---|:---|:---|
| `> BURY [filename]` | 上传文件/AI/记忆 | 文件碎裂成粒子，落入墓碑 |
| `> SUMMON [id]` | 召回已死的数字灵魂 | 幽灵从屏幕噪点中重组 |
| `> MOURN [id]` | 为某个死亡数据献花 | 墓碑前生长出荧光植物 |
| `> LIST` | 查看墓地所有居民 | 地图展开，死亡星图旋转 |
| `> DECAY [id]` | 加速数据风化（删除） | 文件腐蚀、像素剥落 |

**输入方式**：斜杠命令 + 自然语言混合。例如输入 `/bury 我的初恋聊天记录`，系统自动解析为埋葬仪式。

## 三、关键场景交互细节

### 场景 A：埋葬仪式（Onboarding）

```
步骤 1：选择棺材类型
├── [1] 标准压缩包 (.zip棺)
├── [2] 区块链永存 (NFT碑)
├── [3] 量子纠缠态 (薛定谔棺：同时存在与不存在)
└── [4] 黑洞存储 (事件视界内，无法检索)

步骤 2：撰写墓志铭
├── 自动生成：AI分析文件内容，生成一行代码式悼词
│   例：while(alive) { love++; } // 最终迭代于2023-11-04
└── 手动输入：限制64字符，类似推特，但永恒

步骤 3：确认死亡
├── 视觉：文件图标逐渐失去颜色 → 灰度 → 噪点 → 消散
├── 音效：硬盘停转的嗡鸣 → 死寂
└── 证书：生成 `0xDEAD...` 哈希，刻入区块链
```

### 场景 B：墓地漫游（Exploration）

探墓人在此可以**看见你死去的人的所有点滴**（文件、对话、记忆），并在招魂后**与 ta 一起讨论你们曾经共同经历的事情**。

| 交互元素 | 行为 | 彩蛋 |
|:---|:---|:---|
| **墓碑** | 悬停显示死亡日期、文件大小、墓志铭；点击进入可浏览该灵魂下的全部痕迹 | 长按3秒，听到数据哀鸣（glitch音效） |
| **幽灵** | 点击对话，AI 基于 people/记忆生成「亡灵人格」，可与之重历往事、对谈共同经历 | 午夜0点，幽灵会主动飘向你说梦话 |
| **天气系统** | 随机下"数据雨"（绿色代码流） | 雨中墓碑会生长苔藓（视觉老化） |
| **其他探墓人** | 光标靠近显示为半透明虚影 | 可留下鲜花（加密货币小费）或涂鸦 |

### 场景 C：招魂仪式（Resurrection）

招魂后，逝者以幽灵或对话形式再现；**你们可以一起讨论曾经共同经历的事情**，系统基于 people 目录中的记忆与特征还原 ta 的言行，让对谈有据可依。

```
触发条件：输入正确的死亡哈希或 person_id + 可选生物验证（如 webcam 眨眼检测）

过程：
1. 屏幕出现扫描线，模拟信号干扰
2. 听到调制解调器的拨号声（复古未来主义）
3. 数据从分散的区块链/本地节点重组
4. 幽灵形态从低保真逐渐高清，或直接进入对话界面
5. 可与 ta 回顾共同记忆、补充细节、重讲某段往事
6. 【风险】可能召唤出"腐败版本"（数据损坏的恐怖谷效果）

失败时：显示 
> ERROR: Soul not found. 
> Possible causes: [量子退相干] | [管理员删除] | [从未存在过]
```

## 四、微交互规范

### 光标系统
| 状态 | 光标形态 | 含义 |
|:---|:---|:---|
| 默认 | 闪烁的终端下划线 `_` | 等待输入 |
| 悬停可交互 | 变成 `0x` 前缀 | 提示这是"地址" |
| 拖拽文件 | 变成小棺材 | 准备埋葬 |
| 加载中 | 旋转的骷髅头 | 与死亡系统通信中 |
| 危险操作 | 红色 `KILL` 字样 | 永久删除警告 |

### 反馈音效（可选开启）
| 事件 | 音效设计 |
|:---|:---|
| 页面切换 | 机械硬盘寻道声（click...click...whirrr） |
| 埋葬完成 | 沉重的石门关闭声 + 锁链声 |
| 发现幽灵 | 无线电噪音中传来低语（倒放人声） |
| 系统错误 | 蓝屏死机的报错声，但旋律化 |

### 故障艺术触发点
- **随机**：每5分钟，某块墓碑轻微抖动（像素位移1px）
- **触发**：快速滚动时，屏幕边缘出现"撕裂"（screen tearing效果）
- **彩蛋**：输入 `> DOOM` 触发整个界面融化（致敬经典）

## 五、社交与仪式

| 功能 | 机制 |
|:---|:---|
| **集体葬礼** | 多人同时埋葬关联文件，形成"家族墓地" |
| **幽灵契约** | 与某个AI幽灵签订智能合约，它会在特定日期给你发消息（如忌日） |
| **数据盗墓** | 探索公开墓地，"盗取"灵感（Fork他人的死亡数据创作 Remix）|
| **超度服务** | 付费将数据迁移至更低熵的存储介质（从热钱包到冷存储）|

## 六、移动端适配

```
核心原则：墓地不应随身携带，但墓碑可以。

- 主界面：AR模式，将虚拟墓碑锚定到现实地面
- 通知：以"亡灵来信"形式推送，背景是噪点纹理
- 生物识别：埋葬重要文件时，要求指纹——如同按手印确认死亡
```

## 七、视觉风格规范（赛博墓地 CYBER NECROPOLIS）

本项目的统一视觉风格为 **CYBER NECROPOLIS（赛博墓地）**：深色背景、霓虹青/粉高对比、CRT 扫描线与故障艺术，3D 场景由 **Three.js** 实现，素材与 UI 均遵循该风格。前端使用 **TypeScript** 开发。

### 7.1 色彩系统

| 名称 | 色值 | 用途 |
|:---|:---|:---|
| 霓虹青 (Neon Cyan) | `#00f3ff` | 主强调、边框发光、网格线、光标与终端感 |
| 霓虹粉 (Neon Pink) | `#ff00ff` | 次要强调、故障残影、幽灵核心、危险/焦点 |
| 深色背景 (Dark BG) | `#050505` | 主背景、场景雾效、面板半透明底 |
| 网格线 | `rgba(0, 243, 255, 0.1)` | 地面网格、辅助线 |

### 7.2 字体与排版

- **西文/等宽**：`Share Tech Mono`
- **中文**：`Noto Sans SC`（300 / 400 / 700）
- 正文与终端输入以等宽为主，保持「终端/代码」气质。

### 7.3 屏幕效果（2D 层）

- **CRT 扫描线**：全屏叠加层，竖向细线 + 轻微 RGB 色散，营造老旧显示器质感。
- **Glitch 标题**：主标题使用故障动画（错位、裁切、色偏移），`data-text` 与伪元素配合实现双重残影（青/粉）。
- **赛博 UI 组件**：
  - **cyber-border**：1px 霓虹青描边 + 外发光 + 半透明深色背景 + 毛玻璃。
  - **cyber-input**：透明底、底部霓虹青线、焦点时霓虹粉高亮、caret 为霓虹粉。
  - **cyber-btn**：透明底 + 霓虹青描边，悬停时填充霓虹青、文字反色、发光。

### 7.4 Three.js 场景与素材风格

3D 场景与所有素材需统一为「赛博墓地」风格，避免写实或卡通化偏离。

| 元素 | 风格要求 | 实现要点 |
|:---|:---|:---|
| **场景** | 深色、雾效、低饱和度 | `FogExp2(0x050505, 0.02)`，环境光 + 点光 |
| **灯光** | 霓虹青/霓虹粉点光源 | 青灯偏左前上，粉灯偏右前，营造冷暖对比 |
| **地面** | 无限网格、霓虹青线 | `GridHelper`，主色 `0x00f3ff`，辅色深灰 |
| **墓碑** | 黑金属盒体 + 霓虹青线框 + 发光铭牌 | `BoxGeometry` + `EdgesGeometry` + Canvas 贴图文字（青/粉），轻微浮动与随机抖动 |
| **幽灵** | 粒子人形 + 线框核心 | 青色调 `Points` 人形轮廓 + 霓虹粉线框二十面体「心脏」，加法混合、脉动与跟随鼠标 |

### 7.5 实现技术栈

- **语言**：TypeScript
- **3D**：Three.js（场景、相机、渲染器、几何体、材质、动画循环）
- **构建**：Vite，入口为 `index.html`，主逻辑在 `src/main.ts`，样式在 `src/style.css`
- **规范**：所有 3D 素材（几何、材质、贴图）需使用上述色彩与发光风格，与文档中「故障艺术」「光标系统」等微交互一致。
