02 Problem & Solution PRD
Why This Product Exists Digital things die all the time, but we still do not have a ritual for their ending.
That gap is the whole premise of 0xRIP. We already accumulate dead files, broken conversations, deprecated models, and abandoned selves, but we only know how to archive or ignore them. The product proposition is to replace silent accumulation with memorial space, ritual verbs, and closure.
[IMG: Digital Chaos] Problem prompt · click to view/copy full prompt

Problem

Data dies but does not disappear. Files accumulate, conversations fade, models are deprecated, and former selves get abandoned. We only know how to keep the debris or delete it.

[IMG: Digital Graveyard] Solution prompt · click to view/copy full prompt

Solution

A digital graveyard where you bury what matters, summon the dead as data souls, and have one last conversation. Not storage. Not backup. A memorial space with ritual, memory, and closure.

Problem Logic Dead Data Still Haunts

The absence of ritual turns digital endings into silent accumulation rather than meaningful closure.

Product Logic Memorial, Not Storage

The product reframes preservation as burial, presence, mourning, and release instead of file management.

Experience Logic Closure Needs Form

A designed ritual gives people a legible way to return, speak once more, and let memory rest.

05 Project Architecture Engineering
Engineering Thesis The stack is narrow on purpose: one client surface, one agent spine, one durable memory layer.
The browser renders one world and one terminal. Bun owns the request and orchestration loop. SQLite holds durable grave state. The stack stays legible because each layer does one job.
[DIAGRAM: System Architecture Block Diagram] Architecture prompt · click to view/copy full prompt
Frontend One Client Surface

React + Vite + R3F render one graveyard world and one terminal UI. The client is intentionally narrow, with scene, ghost, grave, and terminal as the visible surface.

Backend One Agent Spine

Bun serves the API, validates requests, builds context, runs the GhostAgent loop, and delegates media work through five mmx wrappers.

Storage Two Memory Layers

SQLite holds durable graves, sessions, and promoted memories while the live interaction layer reads and writes shorter-lived session context.

09 Dual Memory System Engineering
Memory Thesis The product needs both fleeting conversation shards and durable promoted memory blocks.
A ghost should remember enough to feel continuous, but not everything deserves permanent storage. That is why 0xRIP separates short-term session memory from promoted long-term archive memory. The user experience gains continuity without pretending that all conversation should last forever.
[DIAGRAM: Dual Memory Architecture — Short-term → Promote → Long-term] Memory prompt · click to view/copy full prompt
Short-Term Live Session Memory

.rip/<id>/sessions/<sid>/memories/*.md stores recent visitor/ghost exchanges and injects the last 10 turns into live context.

Long-Term Promoted Archive Memory

memories in SQLite keeps durable recollections with a 500-memory cap, FIFO eviction, and summarized themes after promotion.

12 Tech Stack Engineering
Build Thesis The stack is pragmatic and sharp-edged: Bun for speed, React for surface, SQLite for memory, MiniMax for presence.
The point of the stack is not novelty. It is coherence. Each layer supports one terminal-first memorial product without extra framework weight or orchestration sprawl.
Runtime + Backend Serve, Validate, Persist
  • Bun runs the server and package workflow with no extra build chain.
  • Bun.serve() keeps the API surface framework-light and explicit.
  • Zod guards request shapes at runtime.
  • SQLite + Drizzle hold graves, sessions, and promoted memory blocks.
Model + Media Reason, Speak, Generate
  • MiniMax M2.7 handles the Anthropic-compatible dialogue path.
  • mmx CLI opens image, music, speech, video, and vision tools.
  • The stack supports one ghost loop and multiple ritual artifact outputs.
Frontend + Experience World, Terminal, Audio
  • React 19 + Vite hold the terminal-first client shell.
  • Three.js + R3F + drei render the graveyard world behind the ritual UI.
  • Zustand and the Web Audio API keep state and ambience lightweight.
Runtime Logic Minimal Server Weight

The backend stays explicit and inspectable instead of vanishing behind a large framework abstraction.

Model Logic One Capability Spine

Dialogue and media generation stay close enough to feel like one product rather than separate integrations.

Experience Logic One Surface, One World

The frontend only needs a terminal surface and a graveyard scene to make the ritual legible.
